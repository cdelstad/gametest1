# Implementation Plan: TypeScript & Instruction Alignment

> Generated by SAGE. This plan is a workspace artifact — update it as implementation progresses.

## Summary

Bring the existing Excalibur.js game codebase into compliance with the project's instruction files (`typescript.instructions.md`, `excalibur.instructions.md`, `architecture.instructions.md`, `error-handling.instructions.md`, `code-comments.instructions.md`). The primary approach is a dependency-ordered sweep: fix project config first (unlocks ES2022 features), remove dead code (reduces noise), establish shared types, then refactor each source file individually. There are 83 identified violations across 18 files, with the `tsconfig.json` being the root-cause enabler for many downstream fixes.

## Technical Context

| Attribute | Value |

| Key Dependencies | Excalibur.js, Lit (for `ExText`), webpack (currently) |

## Watch Out

1. **tsconfig `module` change breaks webpack** — Changing from `commonjs` to `es2022` modules will likely require updates to `webpack.config.js`. The webpack config must use `resolve.extensions` correctly and may need `experiments: { outputModule: true }`. Verify the dev server still works after Phase 1 before proceeding.

2. **`experimentalDecorators` vs TC39 decorators** — The `@customElement` decorator in `ExText.ts` uses the legacy/experimental decorator syntax (stage 2). TypeScript 5.x also supports TC39 decorators (stage 3). Enabling `experimentalDecorators` is correct for the current Lit dependency, but check that Lit's version supports it. Do not enable both `experimentalDecorators` and the new `--experimentalDecorators=false` + `--emitDecoratorMetadata` combo.

3. **`colorRamp.ts` deletion** — Before deleting, verify with `grep -r "colorRamp" src/` that nothing imports it. If something does, that import must be removed in the same task.

4. **`ex-text.js` deletion** — Verify no `<script src="ex-text.js">` in `index.html` or other HTML files. If the JS file is loaded via script tag rather than import, removing it requires updating HTML too.

5. **Config constant renaming cascades** — Renaming `Config.PlayerSpeed` to `Config.PLAYER_SPEED` will break every file that references the old name. All consumers must be updated in the same task or an immediately subsequent one.

6. **`getEntities` case bug is a functional fix** — The one-sided `toLowerCase()` in `utils.ts` isn't just a style issue; it causes entity lookups to silently fail for mixed-case names. This should be prioritized as it affects runtime correctness.

7. **Shared types must match JSON data shapes exactly** — When creating `SpriteSheetConfig` and `AnimationSetConfig` interfaces, they must match the actual shapes in `spritesheets.json` and `animations.json`. Read those files to derive the interfaces, don't guess.

8. **`UIManager._generateUniqueId` removal** — When removing the duplicate and switching to the shared `utils.generateUniqueId()`, verify the implementations are truly identical. If they diverge (different algorithm or seed), pick one and update all callers.

9. **Non-null assertions in lifecycle methods** — `this.scene!` in `Character.ts` is used inside `onInitialize`, where `scene` is guaranteed by Excalibur. Instead of a null guard, use `override onInitialize(engine: Engine)` and access `this.scene` — it is safe in `onInitialize` per Excalibur's lifecycle contract. Document this with a comment rather than adding unnecessary null checks.
