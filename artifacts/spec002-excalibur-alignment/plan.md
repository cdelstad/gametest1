# Implementation Plan: Excalibur Instruction Alignment Fixes

> Generated by SAGE. This plan is a workspace artifact — update it as implementation progresses.

## Summary

Fix 8 code misalignments against the project's `excalibur.instructions.md` across 5 source files. The highest-risk work is a tightly coupled set of 4 fixes in `AnimationManager.ts` (parameter swaps, manual loading, and API modernization) that interact with `Character.ts`. All bugs are currently masked by square 64×64 sprites on an 8×8 grid — they will surface the moment non-square sprites are introduced. The remaining 4 fixes are independent single-file changes.

## Phases

### Phase 1: Independent Single-File Fixes

Four changes on four separate files with zero inter-dependencies.

- [ ] **Task 1.1**: `main.ts` — Replace `antialiasing: false` with `pixelArt: true` *(Item 4)*
  - Files: `src/main.ts`
  - **What to change**: In the `new Engine({...})` options (line ~72), remove `antialiasing: false` and replace with `pixelArt: true`. This enables Excalibur's purpose-built pixel art rendering pipeline (subpixel blending, UV padding 0.25) rather than the blunt antialiasing toggle.
  - **Import note**: No new imports needed — `pixelArt` is a standard `EngineOptions` property.

- [ ] **Task 1.2**: `utils/utils.ts` — Fix portal y-offset using `obj.height` instead of `obj.width` *(Item 9)*
  - Files: `src/utils/utils.ts`
  - **What to change**: In `addPortal()` (line 10), the actor position is:
    ```
    pos: vec(obj.x + obj.width!/2, obj.y + obj.width!/2)
    ```
    The second `obj.width!` should be `obj.height!`:
    ```
    pos: vec(obj.x + obj.width!/2, obj.y + obj.height!/2)
    ```
    This ensures the portal center is correctly computed for non-square Tiled objects.

- [ ] **Task 1.3**: `Character.ts` — Scope input to scene, guard unmapped keys, remove dead imports *(Item 5)*
  - Files: `src/Character.ts`
  - **Changes**:
    1. **Remove dead imports**: Delete the commented-out import on line 2 (`// import {excalInput} from './utils/Excalinput';`) and line 5 (`// import { getSpriteSheetCoord } from './utils/utils';`).
    2. **Scene-scoped input**: In `onInitialize`, change both `engine.input.keyboard.on(...)` calls (lines ~85, ~90) to `this.scene.input.keyboard.on(...)`. This ensures input handlers only fire when the character's scene is active.
    3. **Guard unmapped keys**: In the `"hold"` handler, the line `this.events.emit(bindings.kb[evt.key], this)` will emit `undefined` for unmapped keys. Wrap in a guard:
       ```
       const action = bindings.kb[evt.key];
       if (action) { this.events.emit(action, this); }
       ```
       Apply a similar guard pattern in the `"release"` handler if it references bindings (currently it doesn't — it just zeros velocity, which is fine).

- [ ] **Task 1.4**: `GameScene.ts` / `Character.ts` — Tiled anchor compensation *(Item 7)*
  - Files: `src/Character.ts` (preferred) OR `src/scenes/GameScene.ts`
  - **What to change**: Actors default to `anchor: vec(0.5, 0.5)` (center), but Tiled places objects with top-left origin. Add `anchor: vec(0, 0)` to the Character constructor's `super()` options:
    ```
    super({
        pos,
        width: 16,
        height: 16,
        collisionType: CollisionType.Active,
        name: 'player',
        anchor: vec(0, 0)
    })
    ```
  - **Alternative**: If anchor (0,0) causes unwanted side effects on collision or camera lock, instead offset the position in `GameScene.ts` where Character is created: `vec(player.x + 8, player.y + 8)` (half of 16×16). The implementer should visually verify which approach produces correct Tiled alignment.
  - **NOTE**: Tasks 1.3 and 1.4 both touch `Character.ts`. They are **SEQUENTIAL** — apply 1.3 first, then 1.4. They do not depend on each other logically; the ordering is solely to avoid file conflicts.

> PARALLEL: Tasks 1.1 and 1.2 can run simultaneously with each other AND simultaneously with 1.3/1.4 (no file overlap).
> SEQUENTIAL: Tasks 1.3 → 1.4 (both modify `Character.ts`).

### Phase 2: AnimationManager Overhaul — Sprite Fixes + Loader Pipeline

All four AnimationManager issues must be addressed together because they interact: the sprite dimension swap, the coordinate order swap, the API modernization, and the loader refactor all affect the same code block.

> BLOCKED BY: Task 1.3 and Task 1.4 (both modify `Character.ts`, which this phase also touches)

- [ ] **Task 2.1**: `AnimationManager.ts` — Fix sprite config swap + coordinate order + API modernization *(Items 1, 2, 6)*
  - Files: `src/AnimationManager.ts`
  - **Fix 1 — spriteWidth/Height swap** (lines 12–13): The `SpriteSheet.fromImageSource` grid config has width and height swapped:
    ```
    // CURRENT (wrong):
    spriteWidth: ss.spriteHeight,
    spriteHeight: ss.spriteWidth,
    
    // CORRECT:
    spriteWidth: ss.spriteWidth,
    spriteHeight: ss.spriteHeight,
    ```
    The field names in `spritesheets.json` (`spriteWidth: 64, spriteHeight: 64`) are correct and match the API parameter names. The code was simply passing them cross-wired.

  - **Fix 2 — getSprite coordinate order** (line 28): `SpriteSheet.getSprite(x, y)` uses `(column, row)` order. The code passes `(row, col)` which is backwards:
    ```
    // CURRENT (wrong):
    spriteSheet.getSprite(element[1].row, element[1].col, element[1].config)
    
    // CORRECT:
    spriteSheet.getSprite(element[1].col, element[1].row, element[1].config)
    ```
    The `animations.json` data uses semantically correct field names: `row` means the vertical grid position, `col` means the horizontal grid position. The mapping is `col → x`, `row → y`.

  - **Fix 3 — Use `Animation.fromSpriteSheetCoordinates`** (lines 23–33): Replace the manual frame-building loop and `new Animation({frames})` with the idiomatic API. This also eliminates the `@ts-ignore` on line 27. The refactored animation creation should look like:
    ```
    const frameCoordinates = subentries.map(([_, frame]) => ({
        x: frame.col,
        y: frame.row,
        duration: frame.duration,
        options: frame.config
    }));
    
    target.graphics.add(
        currentElement[0],
        Animation.fromSpriteSheetCoordinates({
            spriteSheet,
            frameCoordinates,
            strategy: AnimationStrategy.Loop
        })
    );
    ```
    **Important**: Verify the default `AnimationStrategy` matches current behavior. The current `new Animation({frames})` defaults to `Loop`. If `fromSpriteSheetCoordinates` has a different default, set it explicitly. Also check that single-frame animations (like idles) work correctly with `Loop` strategy — they should, but verify.

  - **Imports**: Add `AnimationStrategy` to the excalibur import. Remove `Sprite` from imports if no longer used after the refactor (the manual `Sprite` type annotation in the frames array goes away).

- [ ] **Task 2.2**: `AnimationManager.ts` + `Character.ts` — Async load with error handling *(Item 3)*
  - Files: `src/AnimationManager.ts`, `src/Character.ts`
  - Depends on: Task 2.1
  - **Context**: The dynamic `ImageSource.load()` pattern is **intentional** — this is a paper-doll system where sprites are swapped at runtime without scene reloads. Sprites reload when characters change scenes, on game reload, etc. This module will eventually be extracted into a separate npm package. **Do NOT move loading into `Scene.onPreLoad`.**
  - **Problem**: `setupSpriteAnims` calls `characterSS.load().then(...)` — a fire-and-forget async pattern. This causes: (a) a race condition where the character renders before animations are ready, (b) silent failures if the image fails to load, (c) no way for callers to know when loading is complete.
  - **Approach**:
    1. **Make `setupSpriteAnims` an `async` function** that returns `Promise<void>`. Replace the `.load().then(...)` chain with `await characterSS.load()`. The SpriteSheet creation and animation registration that currently live inside the `.then()` callback move to straight-line code after the `await`.
    2. **Add error handling**: Wrap the `await characterSS.load()` in a `try/catch`. On failure, log a meaningful error (e.g., `console.error(\`Failed to load spritesheet: ${ss.sheet}\`, error)`) and re-throw so callers can handle it. Do not silently swallow the error.
    3. **Update `Character.onInitialize`** to `await` the call to `setupSpriteAnims`. Since Excalibur's `onInitialize` can be `async` (the engine awaits the returned Promise), change the method signature to `async onInitialize(engine: Engine)` and use `await setupSpriteAnims(...)`. This guarantees animations are registered before the first frame renders.
    4. **Keep `ImageFiltering.Pixel`** in the `ImageSource` constructor. Dynamic resources created outside the loader pipeline need explicit filtering regardless of the engine-level `pixelArt: true` setting (which applies to the rendering shader, not to individual resource decoding). If the implementer confirms via Excalibur docs that `pixelArt: true` propagates to dynamically loaded resources, the explicit filter can be removed as a follow-up.
  - **Key constraint**: Callers of `setupSpriteAnims` must treat it as async. Any future call sites (scene transitions, sprite swaps, etc.) must `await` the result or handle the Promise appropriately.

> SEQUENTIAL: Task 2.1 must complete before Task 2.2 (both modify `AnimationManager.ts`; 2.2 builds on 2.1's refactored code).

## Watch Out

1. **Square sprites hide everything.** All three swap bugs (spriteWidth↔Height, getSprite row↔col, getSpriteSheetCoord row↔col) are simultaneously invisible because every sprite is 64×64 on an 8×8 grid. After fixing, the animations WILL select different sprite cells. Visual verification is non-negotiable. If animations look wrong after the fix, the issue is in `animations.json` data, not the code.

2. **Animation data was likely authored against the buggy code.** If the developer visually tuned `row`/`col` values in `animations.json` until the animations "looked right" with the swapped parameters, then the data compensates for the bug. Fixing the code would then require re-authoring the animation data. The plan assumes the data field names are semantically correct (`row` = vertical position, `col` = horizontal position), which is the most likely scenario given the clear naming. But if visual testing shows broken animations, check the Mana Seed sprite sheet layout and re-derive the coordinates.

3. **Async refactor timing.** The current `.load().then()` fire-and-forget pattern means the character actor exists before its sprites are ready. Code elsewhere may silently depend on this timing (e.g., the character appearing with default graphics first). After the async/await refactor, `onInitialize` won't resolve until sprites are loaded. This is correct but changes the initialization order — verify no downstream code assumes the character is ready before its sprites load.

4. **`fromSpriteSheetCoordinates` strategy default.** Verify that the animation strategy defaults to `Loop` (matching the current `new Animation({frames})` default). Single-frame animations (idles) should work with `Loop` but consider whether `Freeze` or `End` is more appropriate for non-looping cases.

5. **`getSpriteSheetCoord` in utils.ts is also swapped** (row/col labels are backwards in the return value). This function is currently **unused** (import commented out in Character.ts). The plan does not fix it since it's dead code. If it's ever re-enabled, the swap must be fixed: `row: Math.trunc(ssNum/columns)`, `col: ssNum % columns`.

6. **Anchor (0,0) side effects.** Changing Character's anchor from default (0.5, 0.5) to (0, 0) shifts the collision box and the camera lock point. The camera will lock to the character's top-left corner instead of center. The implementer may need to compensate by adjusting the camera strategy or using `offset` on the camera. Test with the actual Tiled map.

7. **`pixelArt: true` vs `ImageFiltering.Pixel`.** If `pixelArt: true` sets a global pixel filtering default, the explicit `ImageFiltering.Pixel` in `AnimationManager` becomes redundant. If it doesn't (if `pixelArt` only affects the rendering shader, not resource loading), the explicit filter must remain. Check the Excalibur source or docs during implementation.
